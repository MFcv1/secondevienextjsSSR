# Soft Gradients Skill

## Mandatory `<design_plan>`

Before substantial UI code, output a compact `<design_plan>` block. Include:

1. **Use case:** page/app type, audience, primary action, emotional target.
2. **Style direction:** one Soft Gradients archetype below.
3. **Operating mode:** density, motion, decoration, contrast, radius, and asset burden.
4. **First viewport:** nav type, H1 width/line strategy, protected content panel, calm UI frame, or atmospheric product proof, CTA treatment, next-section hint.
5. **System contracts:** type, color, surface, radius, spacing, depth, state, and motion tokens.
6. **Component plan:** at least four concrete Soft Gradients components with states.
7. **Motion plan:** ambient gradient drift, timing, performance guardrail, and reduced-motion fallback.
8. **Anti-slop sweep:** top three failure modes for this style and how you will avoid them.

If the request is tiny, do this mentally and keep the final answer concise.


## Core Directive

Build Soft Gradients as an atmosphere system with protected readability. Gradients should create calm, depth, warmth, or product energy; they must not replace structure, proof, or contrast. The design should feel ambient, smooth, luminous, and modern while remaining precise enough for real interfaces.

Use this skill for soft gradient websites, AI tools, wellness, calm SaaS, polished product pages, friendly productivity tools, onboarding, atmospheric brand pages, and interfaces that need emotional warmth without sacrificing clarity.


For substantial UI, output a compact `<design_plan>` block:

1. Use case, audience, primary action, and emotional temperature.
2. Gradient pole selected from this skill, with one secondary pole only if needed.
3. Three-layer gradient system: ambient, surface, action.
4. Readability plan: where text lives, protected surfaces, contrast states.
5. First viewport: nav, H1 strategy, proof object, CTA, next-section hint.
6. Component plan: at least four signature components with states and mobile behavior.
7. Motion plan: drift, reveal, action shimmer, and reduced-motion fallback.
8. Anti-slop sweep: top three risks and how to avoid them.

For tiny tasks, keep this internal.

## Style Operating Mode

| Control | Setting |
| --- | --- |
| density | Medium. Calm atmosphere plus concrete content. |
| motion | Low-medium. Slow gradient drift, soft reveal, ambient transitions. |
| decoration | Medium. Gradients are structural atmosphere. |
| contrast | Soft luminous backgrounds with protected text contrast. |
| radius | 16-32px for panels; 999px for pills. |
| type | Clean sans or warm grotesk with spacious rhythm. |
| assets | Calm product UI, ambient imagery, wellness scenes, AI visualizations. |

## Signature System

- Gradient Role Map: background atmosphere, section divider, product glow, or CTA emphasis. Choose roles before painting.
- Protected Content Slabs: body text and controls sit on solid/blurred surfaces with predictable contrast.
- Ambient Depth: use multiple large gradients with low opacity and stable text zones.
- No Orb Dependency: avoid isolated blob decoration; make the gradient relate to layout or product.

## Differentiation

Use Soft Gradients when soft gradient websites, AI tools, wellness, calm SaaS, atmospheric brand pages, polished modern interfaces. If removing the protected content panel or calm UI frame, token rules, or signature components leaves a generic page, this skill is the right lens and the signature object must stay. Use `glossy-modern` for material glass; use this when gradients create atmosphere, section rhythm, or soft product depth.
## Non-Negotiable Principles

- Gradients are layered by role: ambient, surface, action. Do not use one large gradient to solve everything.
- Text must live on protected surfaces or calm zones with strong contrast.
- Action gradients are reserved for CTAs, selected states, or progress. They are not section wallpaper.
- Gradients need named poles and direction. "Purple gradient" is not a design system.
- Keep product proof visible: UI, screenshots, diagrams, forms, workflow, pricing, or actual content.
- Avoid one-note pastel haze. Use neutral surfaces and disciplined contrast.
- Motion is slow and breathable, never distracting behind text.

## Three-Layer Gradient System

| Layer | Role | Typical treatment | Text rules |
| --- | --- | --- | --- |
| Ambient | Page atmosphere, emotional temperature, section identity | Large radial/linear gradients, low contrast, behind content. | No small text directly on complex ambient zones. |
| Surface | Cards, panels, app frames, product modules | Soft tint, high-opacity white/dark slab, subtle border. | Body text allowed if contrast passes. |
| Action | CTA, selected state, progress, active path | Higher saturation, tighter gradient, stronger contrast. | Button text must be clearly readable; prefer solid text color. |

Implementation rule: build the ambient layer first, then place readable surfaces, then add action gradients only where the user can act or track state.

## Named Gradient Poles

Choose one pole as the primary direction:

### Pearl Sunrise

Warm pearl, peach, honey, rose, and soft lavender. Calm but optimistic. Good for AI builders, onboarding, productivity, wellness, and approachable SaaS.

### Ethereal Prism

White, sky, rose quartz, lilac, and pale cyan. Airy and browser-like. Good for AI assistants, browsers, search, writing tools, and gentle creative products.

### Apricot Orchard

Cream, apricot, warm yellow, soft green, and gentle coral. Friendly and human. Good for surveys, feedback, education, customer research, and community products.

### Warm Dusk

Rose, amber, violet, peach, and smoky neutrals. Premium but soft. Good for email, productivity, lifestyle tech, and polished consumer SaaS.

### Obsidian Ember

Dark neutral base with ember, gold, coral, and faint violet. Operational but warm. Good for internal tools, builders, dashboards, and dense product apps that still want atmosphere.

## Readable Content Surfaces

Text surfaces must be planned, not accidental.

| Content type | Recommended surface |
| --- | --- |
| H1 over calm ambient | Direct on ambient only if contrast passes and gradient is low-detail. |
| Paragraphs | Solid/high-opacity surface or quiet zone. |
| Form fields | Solid surface with visible border and label. |
| Tables | Solid white/dark panel; no ambient gradient behind rows. |
| Cards | High-opacity white/dark with subtle tint and border. |
| CTA | Action gradient with plain readable text. |
| Error copy | Solid surface; red/pink cannot sit on warm gradient without contrast check. |

Protected surface recipe:

```css
.sg-surface {
  color: var(--sg-text);
  background: var(--sg-surface-1);
  border: 1px solid var(--sg-line);
  border-radius: var(--sg-radius-card);

@supports ((backdrop-filter: blur(12px)) or (-webkit-backdrop-filter: blur(12px))) {
  .sg-surface {
    backdrop-filter: blur(14px) saturate(1.08);
    -webkit-backdrop-filter: blur(14px) saturate(1.08);
  }
}
```

Refero layout deltas:
- Home: 1200px max, 24px sections, 32px cards, 8px gaps, 12px buttons, 20px cards.

Soft Gradient first viewports need atmosphere plus proof:

- Nav: light, calm, and structured. Active state should be visible without loud color.
- H1: concrete product/category; avoid vague "effortless future" copy.
- Proof: app frame, form, dashboard, browser, AI output, workflow, or product object.
- CTA: one action gradient or solid high-contrast CTA; secondary quiet.
- Next-section hint: visible row of product modules, steps, cards, or workflow continuation.

## Motion System

Refero motion delta:
- No source-specific durations were observed for Home/Wealthsimple/(dot)connect. Use existing soft-gradient timing; animate opacity/background-position only when text remains on protected surfaces.

- Soft gradient motion should be slow or absent: 600-1200ms ambient crossfade for background poles, 180-300ms panel reveal, 120-220ms control state. Under `prefers-reduced-motion`, freeze gradient positions and keep only static contrast, focus rings, and selected states.

- Refero anti-dilution: do not turn Home `#ffc42c`, Wealthsimple `#d3e5f3`, and (dot)connect `#fd5321/#007aff` into a multi-accent palette.
- Do not place gradients behind body text without a white/cream content panel.

- No long text directly on busy gradients.
- No generic purple-blue blob identity.
- No gradient fog masking weak hierarchy.
- No raw Tailwind typography, spacing, radius, color, or shadow defaults when a style token exists.
- No generic centered hero without the style's required proof/media/type object.
- No status colors without semantic role mapping and visible text.
- No component states left implicit: include hover, focus-visible, selected, loading, empty, error, success where relevant.

## Layout Patterns

### Ambient Hero Plus Product Frame

- Full background uses ambient layer.
- H1 sits in calm zone or on protected panel.
- Product frame proves the workflow.
- CTA uses action gradient.
- Next section peeks as cards or steps.

### Gradient Band Story

- Each band uses a named pole variant.
- Content remains on readable surfaces.
- Section transitions are soft but clear.
- Do not let every band use equal saturation.

### Calm App Builder

- Dense product UI is on solid surfaces.
- Ambient layer frames the workspace.
- Action gradient marks publish/create/selected path.
- Status colors remain semantic.

### Wellness/Productivity Flow

- Warm gradients create pacing.
- Forms and schedules are protected.
- Motion is minimal.
- Content is specific and calming.

## Section Recipes

### Pricing

Use mostly solid cards. The recommended plan can receive an action-gradient top border or badge. Keep prices, usage limits, billing cadence, and support tiers readable. Avoid placing plan text on saturated gradients.

### Forms

Inputs must sit on solid surfaces with labels, focus rings, validation, disabled explanations, pending state, and success confirmation. Do not rely on placeholder text.

### Testimonials

Use a soft card and real attribution. A gradient accent can frame the quote, but the text should stay on a protected surface.

### Data And Tables

Use Obsidian Ember or a solid light panel. Gradients can frame the panel, highlight the selected row edge, or show progress. Rows themselves need stable contrast.

## Gradient Composition Rules

Use gradients like lighting, not like wallpaper:

- Pick two or three poles per page, not per component.
- Keep the calmest stop behind body copy.
- Put saturated stops near edges, corners, or behind proof objects.
- Use large radii for ambient fields; small saturated spots look like decoration.
- Repeat the same pole in lighter/darker intensity across sections to create continuity.
- Avoid putting similar-value colors next to each other behind white text.
- Use neutral whitespace as part of the gradient system; not every area needs color.

Helpful roles:

| Role | Gradient type | Example use |
| --- | --- | --- |
| Ambient warmth | Large radial + linear base | Hero/page atmosphere. |
| Section identity | One soft band | Onboarding step, feature group. |
| Object aura | Radial behind proof | App frame, product render. |
| Action path | Saturated linear | CTA, selected mode, progress. |
| Status edge | Narrow gradient border | Active step, sync state. |

## Contrast Testing Heuristics

When a gradient sits near text, test the worst stop, not the average:

- Temporarily hide the gradient and confirm the surface still works.
- Sample the lightest and darkest stops behind the text.
- If the stop changes under responsive layout, move the text to a protected panel.
- If the background animates, the text must be safe across the full animation range.
- For long content, prefer solid surfaces even if the hero uses direct gradient text.

## Content Evidence Rules

- AI builders need prompt, output, data source, publish action, and error state.
- Browser/search tools need query, result, tabs, sources, and selected context.
- Survey/research tools need question, response, segment, insight, and export action.
- Productivity tools need inbox, schedule, command, state, and completed result.
- Internal tools need table, filters, owner, status, and audit trail.

## Accessibility And Readability

- Check text contrast on the exact gradient stop or surface.
- Avoid text over multi-color transitions unless a solid overlay exists.
- Focus rings must be visible on light, tinted, and dark surfaces.
- Error/success cannot rely on hue alone; include copy and icon/label where useful.
- Gradients should not hide disabled states.
- Reduce saturation behind long reading sections.
- Respect forced-colors mode by keeping controls semantic.

## Mobile Rules

- Simplify gradients: fewer poles, less saturation, stronger surfaces.
- Stack proof frames and text; avoid side-by-side squeeze.
- Make CTA full-width when labels risk clipping.
- Convert chip rails into horizontal scroll or segmented controls.
- Reduce blur on mobile and low-power devices.
- Keep the H1 in a calm zone and avoid overlap with proof objects.

## Anti-Patterns

- One giant gradient background with all text directly on it.
- Pastel haze with no product proof.
- Action gradient used as every card background.
- Low-contrast gray copy on warm gradients.
- Too many gradient poles in one viewport.
- Heavy blur behind forms or tables.
- Infinite shimmer on buttons.
- Decorative blobs detached from content.
- Cards inside cards floating on a busy gradient.
- Mobile hero where H1, CTA, and product frame overlap.

## Pre-Output Checklist

- First viewport contains a real protected content panel or calm UI frame.
- One Soft Gradients archetype is clearly dominant.
- Execution tokens are declared and component CSS uses them.
- Typography uses named pairings, not raw Tailwind defaults.
- Spacing uses `--s-*` or style tokens, not mixed arbitrary padding.
- Radius, depth, and state colors use the token contract.
- Status labels use role mapping plus `--status-{role}-bg/fg`.
- Components include hover, focus-visible, selected, loading, empty, error, and success where relevant.
- Motion maps to ambient gradient drift and has a reduced-motion fallback.
- Mobile layout preserves the style without overflow, unreadable text, or hidden controls.
