<a id="philosophy"></a>
## 1. CORE PHILOSOPHY

Minimal Design is not the absence of design. It is the absence of unearned
visual decisions.

Minimalism works when the page feels:

- clear
- deliberate
- calm
- structured
- honest
- readable
- useful
- proportioned
- confident

It fails when it feels:

- empty
- generic
- underdesigned
- low contrast
- weakly branded
- sterile
- lifeless
- hard to scan
- like a wireframe

The central rule:

**Remove decoration, not hierarchy.**

Minimalism should make hierarchy more obvious, not less. If removing color,
shadows, imagery, or chrome makes the page confusing, then those elements were
doing necessary work and must be replaced by better structure:

- type scale
- spacing
- alignment
- surface contrast
- accent discipline
- grid logic
- copy clarity
- product proof

The source set gives five poles:

| Source | Archetype | What it teaches |
| --- | --- | --- |
| General Intelligence Company | Architectural minimal authority | One atmospheric hero, then restrained light UI and calm blue accent |
| Sprig | White canvas thoughtful function | Soft SaaS minimalism, warm gradients as atmosphere, navy hierarchy |
| Standards | High-contrast precision blueprint | Hard flat minimalism, black/gray/orange, no shadows, strict type |
| Rox | Analytical blueprint on pure white | Giant hero type, data clarity, single blue CTA, subtle status colors |
| Copy | Strategic polished steel | Enterprise minimal SaaS, violet action, 4px radii, no-shadow hierarchy |

Together they define a broad minimal vocabulary:

- soft minimal
- hard minimal
- architectural minimal
- analytical minimal
- enterprise minimal
- technical minimal

Minimal design should never be reduced to "white background, black text,
small gray subtitle." The skill is in choosing exactly which elements are
allowed to speak.

### The Minimal Stack

Build minimal UI from these layers:

1. **Canvas** - the emotional ground: off-white, pale gray, dark navy, or black.
2. **Text hierarchy** - the main design engine.
3. **Spacing** - the rhythm that separates ideas.
4. **Surface** - subtle cards, panels, or no cards.
5. **Accent** - one color role, usually action.
6. **Product proof** - screenshots, UI frames, diagrams, or actual workflow.
7. **Interaction** - clear states without ornamental animation.

If you skip text hierarchy or spacing, minimal design becomes empty. If you
skip product proof, it becomes vague. If you overuse accent, it becomes
ordinary SaaS.

### Minimal Design Laws

1. Use fewer colors, but stronger roles.
2. Use fewer typefaces, but more exact type decisions.
3. Use fewer surfaces, but clearer levels.
4. Use fewer shadows, but better borders and spacing.
5. Use fewer words, but more concrete copy.
6. Use fewer illustrations, but more product evidence.
7. Use fewer components, but stronger states.
8. Use one primary accent, not five weak ones.
9. Let whitespace group meaning.
10. Make every visual element answer "why am I here?"

## 2. WHEN TO USE THIS SKILL

Use Minimal Design when the user asks for:

- minimal
- minimalist
- clean
- simple
- elegant
- precise
- restrained
- premium but quiet
- no clutter
- product-first
- typography-led
- neutral palette
- architectural UI
- clean SaaS
- modern interface
- calm product page
- clear dashboard
- serious B2B page

It works especially well for:

- B2B SaaS
- AI tools
- finance automation
- research tools
- infrastructure products
- enterprise software
- design/dev tools
- docs and changelogs
- analytics pages
- product-led landing pages
- pricing pages
- dashboards
- onboarding
- internal tools

Use with caution for:

- games
- youth consumer products
- highly emotional campaign pages
- music/fashion/editorial brands needing strong expression
- products that need playful illustration
- pages that depend on photography

Avoid this skill when:

- the user explicitly wants maximalism
- brand identity depends on vibrant illustration
- the page is a campaign requiring spectacle
- the product needs strong entertainment value
- minimalism would make the category feel cold or untrustworthy

Pair with:

| Pairing | Result |
| --- | --- |
| Technical Sans | More product/tool precision |
| Editorial Minimal | More narrative and typographic grace |
| High Contrast | Sharper, harder, more exacting |
| Clean SaaS | More commercial B2B usability |
| Soft Gradients | Softer atmosphere without clutter |
| Monochrome UI | More disciplined grayscale |
| High-End Design | More luxury restraint |

<a id="archetypes"></a>
## 3. THE 5 SOURCE ARCHETYPES

### 3.1 General Intelligence Company - Architectural Minimal Authority

**Mood:** calm authority, advanced technology, architectural restraint.

This archetype opens with a dark, atmospheric hero but keeps the core product
UI minimalist and architectural. It uses a serif/sans type relationship, mostly
achromatic sections, soft shadows, translucent overlays, and a single cool blue
accent.

Key moves:

- one dark hero only
- light contained sections after hero
- centered headline
- subtle card shadows
- cool blue action/focus
- geometric/abstract content visuals
- no lifestyle photography

Best for:

- AI companies
- research labs
- advanced technology
- executive B2B
- sophisticated product launches

### 3.2 Sprig - White Canvas Thoughtful Function

**Mood:** soft, clear, functional, warm.

Sprig is minimalism that does not feel cold. It uses white and warm cream,
deep navy text, slate copy, soft gradients for atmosphere, muted borders, and
large radii on imagery. The interface remains product-led and information-rich.

Key moves:

- warm/white canvas
- deep navy hierarchy
- charcoal CTA
- muted borders
- gradients as background mood only
- generous vertical spacing
- compact product modules

Best for:

- customer research
- analytics
- feedback tools
- B2B SaaS needing warmth
- AI insight products

### 3.3 Standards - High-Contrast Precision Blueprint

**Mood:** exacting, flat, sharp, typographic.

Standards represents hard minimalism. Near-white canvas, black text, hairline
dividers, no shadows, no gradients, minimal orange action. Cards may have
0px radius and no elevation. It is clean because it refuses softness.

Key moves:

- Canvas Ice background
- black primary text
- one orange accent
- no shadows
- hairline dividers
- same typeface throughout
- tight letter spacing
- flat grids

Best for:

- design systems
- standards/docs
- precision brands
- research sites
- technical reference
- product pages needing authority

### 3.4 Rox - Analytical Blueprint on Pure White

**Mood:** analytical, financial, spacious, high-clarity.

Rox uses a pale page canvas, white surfaces, giant display type, geometric sans
body text, subtle blue CTA, and barely-there elevation. It is minimal but not
plain: the hero typography is dramatic, while the UI remains restrained.

Key moves:

- huge display headline
- off-white page canvas
- single blue CTA
- compact neutral cards
- status colors only as data tags
- subtle shadows
- clear product screenshot framing

Best for:

- finance automation
- analytics
- AI ops
- B2B platforms
- data products

### 3.5 Copy - Strategic Polished Steel

**Mood:** enterprise, structured, muted, conversion-focused.

Copy shows minimal B2B SaaS with a strong action accent. It uses a cloud-gray
canvas, near-black text, slate borders, violet CTAs, strict 4px radii, no
shadow-based elevation, and precise spacing.

Key moves:

- Cloud Burst page/card background
- Midnight Ink text
- Slate Echo borders
- Violet Impulse CTA only
- 4px radius
- no shadows
- strong display type
- process/flow visuals

Best for:

- AI SaaS
- enterprise software
- workflow automation
- sales/marketing platforms
- conversion pages

<a id="shared-dna"></a>
## 4. SHARED DESIGN DNA

### 4.1 One accent has one job

Minimal systems usually allow one chromatic accent:

| Source | Accent | Role |
| --- | --- | --- |
| GIC | blue | interactive/featured |
| Sprig | charcoal/navy with soft gradients | action and hierarchy |
| Standards | orange | primary action only |
| Rox | blueprint blue | CTA and active state |
| Copy | violet | CTA and active state |

The accent should not become decoration. It should answer:

- where do I click?
- what is active?
- what is important?
- what changed?

### 4.2 Neutrals do most of the work

Minimal design depends on neutral precision:

- text contrast
- background contrast
- border visibility
- disabled states
- muted copy
- surface levels

Bad minimalism uses one pale gray for everything. Good minimalism has a neutral
scale with clear jobs.

### 4.3 Typography is the main visual system

Minimal pages do not have much decoration, so type carries identity:

- one display font for hero
- one sans for body/UI
- controlled tracking
- restrained weights
- strong line-height decisions
- enough contrast between heading/body/caption

If typography is generic, the page feels generic.

### 4.4 Spacing is content architecture

Whitespace is not empty. It groups, ranks, and paces information.

Good spacing:

- creates section rhythm
- separates topics
- makes CTAs obvious
- lets product visuals breathe
- supports scanning

Bad spacing:

- creates huge blank gaps with no meaning
- makes everything feel disconnected
- uses equal spacing for unequal hierarchy

### 4.5 Product proof matters

Minimal design should show the product clearly:

- screenshots
- UI cards
- workflow panels
- diagrams
- tables
- forms
- dashboard details

If minimal design does not include product proof, it risks feeling like an
empty manifesto.

### 4.6 Decoration is replaced by rules

Minimal design removes:

- decorative gradients
- unnecessary illustrations
- multiple accents
- heavy shadows
- arbitrary cards
- extra borders

It replaces them with:

- exact type scales
- alignment
- neutral palettes
- consistent radii
- surface levels
- strong copy
- product examples

<a id="not-empty"></a>
## 5. MINIMALISM IS NOT EMPTINESS

Minimal design often fails because teams confuse removal with refinement. A
blank white page with a small headline is not refined by default.

### 5.1 What Minimalism Removes

- decorative clutter
- redundant colors
- vague illustrations
- too many shadows
- repeated cards with no hierarchy
- unnecessary icons
- ornamental gradients
- meaningless animations
- extra font families

### 5.2 What Minimalism Must Keep

- clear headline
- clear CTA
- enough contrast
- readable body copy
- defined sections
- product proof
- interaction states
- visual rhythm
- brand clue
- accessible focus states

### 5.3 Minimal Content Density

Minimal does not always mean sparse. It can be dense when the density is
organized:

- compact changelog
- dashboard card
- feature comparison
- pricing table
- form
- product screenshot
- workflow diagram

The rule is:

```txt
Sparse outside, precise inside.
```

Use air around the module. Inside the module, align and compact.

### 5.4 Minimal Brand Signal

A minimal brand can still be recognizable through:

- one accent color
- one typeface detail
- one radius rule
- one surface treatment
- one hero rhythm
- one image style
- one CTA shape

Do not remove all brand signal. Remove weak brand signal.

### 6.7 Accent Rules

| Accent | Good role | Bad role |
| --- | --- | --- |
| Blue | CTA, active, link | every headline, every icon |
| Orange | primary action | decorative section color |
| Violet | CTA, active nav, process emphasis | body text, background filler |
| Charcoal | CTA on soft canvas | muted body text if too low contrast |
| Gradients | atmosphere in imagery | functional UI state |

### 6.8 Semantic Colors

Minimal systems should not confuse brand accent with semantic states:

- red = error/destructive
- orange/amber = warning
- yellow = notice/information
- green = success
- blue/violet = brand/action only unless explicitly semantic

Keep semantic colors small:

- tags
- badges
- status dots
- alert borders
- chart marks

Do not let semantic colors become a decorative palette.

<a id="typography"></a>
## 7. TYPOGRAPHY SYSTEMS

Minimal design lives or dies by typography.

### 7.1 Typeface Models

#### Single Sans System

Use for Standards-like precision.

- one typeface
- two weights
- consistent tracking
- no serif/display companion

Best for:

- hard minimal
- documentation
- standards pages

#### Serif Display + Sans Body

Use for GIC/Rox-like authority.

- serif or distinctive display for hero only
- sans for body and UI
- large scale and tight line-height

Best for:

- AI authority
- finance
- analytical products

#### Custom Display + Inter UI

Use for Copy-like enterprise minimal.

- custom display for marketing headlines
- Inter for functional text
- strict letter spacing

Best for:

- B2B SaaS
- AI workflow
- enterprise conversion

### 7.2 Type Scale

Minimal pages need deliberate scale:

| Token | Size | Use |
| --- | --- | --- |
| caption | 10-12px | metadata, tags |
| label | 13-14px | nav, form labels |
| body-sm | 14px | compact text |
| body | 16px | paragraphs |
| body-lg | 18-20px | lead paragraphs |
| heading-sm | 20-24px | card headings |
| heading | 28-36px | section headings |
| heading-lg | 40-56px | major sections |
| display | 64-88px | hero |
| statement | 106-183px | rare analytical hero |

### 7.3 Tracking

Minimal type often uses subtle negative tracking:

- body: 0 or slight negative only if font supports it
- display sans: -0.01em to -0.02em
- display serif: tight line-height, not always tracking
- small labels: 0 or slight positive if all caps

Do not use random tracking per component. Make it systematic.

### 7.4 Weight Rules

- Avoid too many weights.
- Use 400 for body.
- Use 500/600 for headings or key labels.
- Avoid default bold everywhere.
- If using a display typeface, reserve it for major moments.

### 7.5 Line Height

| Text | Line height |
| --- | --- |
| huge hero | 0.8-1.0 |
| display | 1.0-1.15 |
| heading | 1.15-1.3 |
| card heading | 1.2-1.4 |
| body | 1.45-1.65 |
| compact UI | 1.2-1.4 |

### 7.6 Typographic Anti-Slop

Reject:

- body copy too pale
- heading and body too similar
- giant headline with no useful subcopy
- generic Inter for every hero when brand needs distinction
- too many font families
- default browser line-height
- viewport-scaled type with no min/max

Replace with:

- one clear type system
- enough scale contrast
- precise line-height
- tight but readable tracking
- defined display font role

<a id="spacing"></a>
## 8. SPACING, SHAPE, AND DENSITY

### 8.1 Base Unit

Use a 4px base unit for most minimal systems.

```css
:root {
  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-5: 20px;
  --space-6: 24px;
  --space-8: 32px;
  --space-10: 40px;
  --space-12: 48px;
  --space-16: 64px;
  --space-18: 72px;
  --space-20: 80px;
  --space-30: 120px;
}
```

Use 8px base only if the source style has broader rhythm and fewer compact UI
elements.

### 8.2 Density Modes

#### Compact Precision

- section gap: 24-48px
- card padding: 12-16px
- element gap: 4-12px
- small radius: 0-6px
- no decoration

Use for:

- docs
- changelogs
- precision brands
- standards pages

#### Comfortable SaaS

- section gap: 64-80px
- card padding: 24-32px
- element gap: 12-20px
- radius: 8-16px
- subtle borders

Use for:

- B2B SaaS
- AI products
- conversion pages

#### Architectural Spacious

- section gap: 96-140px
- card padding: 32-80px
- element gap: 16-32px
- strong typographic scale
- few cards

Use for:

- premium product pages
- AI labs
- finance hero pages

### 8.3 Radius Systems

| Style | Radius |
| --- | --- |
| hard precision | 0-4px |
| analytical | 6-12px |
| enterprise | 4px strict |
| soft SaaS | 12-24px |
| architectural | 12-24px and pill controls |

Rules:

- choose a small set of radii
- do not mix random 7/13/19px values
- use pill only when it is part of the grammar
- keep form fields predictable

### 8.4 Alignment

Minimalism amplifies alignment mistakes.

Rules:

- align text columns
- align card edges
- keep grid gutters consistent
- avoid arbitrary floating elements
- center only when content is truly singular
- use left alignment for dense content

### 8.5 Section Spacing

Use spacing to show relationship:

- 8-12px: label to heading
- 16-24px: heading to paragraph
- 24-40px: paragraph to CTA
- 48-80px: heading block to product visual
- 72-120px: section to section

Avoid equal spacing everywhere. Equal spacing makes hierarchy flat.

## 9. SURFACE AND ELEVATION SYSTEMS

Minimal surface systems should be quiet and consistent.

### 9.1 Flat Hairline System

Use for Standards-like hard minimalism.

```css
.hairline-card {
  background: transparent;
  border: 0;
  border-top: 1px solid #d7d7d7;
  border-left: 1px solid #d7d7d7;
  border-radius: 0;
}
```

### 9.2 Subtle White Card

Use for Rox-like analytical minimalism.

```css
.minimal-card {
  background: #ffffff;
  border: 1px solid #f0efef;
  border-radius: 6px;
  box-shadow: 0 2px 4px rgba(0,0,0,.06);
}
```

### 9.3 No-Shadow Enterprise Card

Use for Copy-like B2B minimalism.

```css
.enterprise-card {
  background: #f6fafb;
  border: 1px solid #e4edf1;
  border-radius: 4px;
  box-shadow: none;
}
```

### 9.4 Soft Functional Card

Use for Sprig-like minimalism.

```css
.soft-card {
  background: #ffffff;
  border: 1px solid #e2e8f0;
  border-radius: 16px;
  box-shadow: 0 18px 50px rgba(15, 23, 42, .06);
}
```

### 9.5 Translucent Architectural Card

Use for GIC-like authority.

```css
.architectural-overlay {
  background: rgba(222,226,222,.16);
  border: 1px solid rgba(255,255,255,.16);
  border-radius: 24px;
  backdrop-filter: blur(8px);
  box-shadow: none;
}
```

### 9.6 Elevation Rules

Use:

- borders first
- background shifts second
- shadows third
- blur rarely

Avoid:

- heavy drop shadows
- nested cards
- shadow on every component
- gradient cards unless gradient is source-defined

### 9.7 Surface Count

Most minimal pages need 3-4 surface levels:

| Level | Use |
| --- | --- |
| Canvas | page background |
| Subtle fill | badge/input/section wash |
| Surface | card/panel |
| Accent surface | CTA/active state |

If you have more than 5 surface levels, the system is probably too complex.

## 13. MOTION AND INTERACTION

Minimal motion should confirm, not entertain.

### 13.1 Principles

- fast micro-interactions
- subtle hover states
- clear focus states
- no decorative looping unless atmosphere is part of source style
- no bounce unless explicitly brand-driven
- no slow reveal that delays content

### 13.2 Timing

| Interaction | Duration |
| --- | --- |
| Link hover | 120-160ms |
| Button hover | 140-180ms |
| Card hover | 160-220ms |
| Input focus | 120-180ms |
| Section reveal | 220-360ms |

### 13.3 Hover States

```css
.minimal-card {
  transition: border-color 160ms ease, box-shadow 160ms ease, transform 160ms ease;
}

.minimal-card:hover {
  border-color: color-mix(in srgb, var(--accent) 20%, var(--border));
}
```

If the source style forbids shadows, do not add shadow on hover. Use border or
background shift.

### 13.4 Focus States

```css
.minimal-focus:focus-visible {
  outline: 0;
  box-shadow: 0 0 0 3px color-mix(in srgb, var(--accent) 16%, transparent);
  border-color: var(--accent);
}
```

### 13.5 Reduced Motion

```css
@media (prefers-reduced-motion: reduce) {

    animation-duration: .001ms !important;

  }
}
```

## 14. ANTI-SLOP RULES

### 14.1 Empty Minimal Slop

Reject:

- giant blank hero with vague headline
- no product screenshot
- weak CTA
- pale gray body text
- no visual hierarchy

Replace with:

- strong headline
- clear CTA
- product proof
- readable text
- intentional spacing

### 14.2 Generic SaaS Minimal Slop

Reject:

- white background + blue button + 3 cards
- generic bento grid
- stock icons
- same radius everywhere
- no brand clue

Replace with:

- source-specific accent
- strict type system
- product-specific layout
- unique surface/radius rule

### 14.3 Low Contrast Slop

Reject:

- light gray body copy
- pastel CTA text
- disabled-looking nav
- border too faint to group content

Replace with:

- strong neutral scale
- readable secondary text
- focus/active state clarity

### 14.4 Overdecorated Minimal Slop

Reject:

- decorative gradients in every section
- many accent colors
- multiple display fonts
- heavy shadows
- unrelated illustrations

Replace with:

- one accent
- one type system
- one surface grammar
- one product proof style

### 14.5 Card Slop

Reject:

- every section as floating card
- cards inside cards
- repeated equal cards with no hierarchy
- shadows on all cards

Replace with:

- section bands
- grid/list hierarchy
- borders/background shifts
- one featured card if needed

### 15.1 Choose Archetype

If the product is advanced AI/research:

- choose Architectural Minimal Authority

<a id="prompting"></a>
## 21. PROMPTING GUIDE

### 21.1 General Minimal Prompt

Create a minimal product interface where hierarchy, typography, spacing, and
product proof do the work. Use a restrained neutral palette, one primary
accent color, clear CTA, subtle surfaces, and consistent radii. Avoid decorative
gradients, random illustrations, low-contrast gray text, unnecessary cards, and
empty white space with no information structure.

### 21.2 Architectural Minimal Prompt

Design a sophisticated architectural minimal product site with one dark
atmospheric hero, then restrained light sections. Use serif display headings,
clean sans body text, achromatic surfaces, soft shadows, one cool blue accent
for interactive elements, and contained product/abstract visuals.

### 21.3 Soft Functional Prompt

Design a soft minimalist SaaS page on a white/warm canvas with deep navy
headings, slate body copy, charcoal primary CTA, muted borders, generous
vertical spacing, compact product modules, and subtle warm gradients only in
imagery or section atmosphere.

### 21.4 High-Contrast Precision Prompt

Design a hard minimal precision page with near-white canvas, black text,
hairline gray dividers, one orange CTA, no shadows, no gradients, strict grid
alignment, and a single sans typeface with controlled letter spacing.

### 21.5 Analytical Blueprint Prompt

Design an analytical minimalist product page with pale off-white canvas, white
cards, giant display headline, crisp sans body text, one blueprint-blue CTA,
subtle status chips only for data, and very light shadows or borders.

### 21.6 Enterprise Violet Prompt

Design a restrained enterprise SaaS page with cloud-gray canvas, near-black
text, slate borders, one violet CTA/active-state accent, strict 4px radius,
no shadows, geometric process visuals, and precise spacing.

### 21.7 Negative Prompt

Do not use random decorative gradients, stock illustrations, multiple accent
colors, weak pale gray body text, floating cards everywhere, generic SaaS bento
layouts, heavy shadows, or minimal layouts that lack product proof and clear
conversion hierarchy.

<a id="pattern-library"></a>
## 22. MINIMAL DESIGN PATTERN LIBRARY

This section turns the source observations into reusable patterns. Use it as a
production menu: pick one primary page pattern, two or three supporting modules,
and one interaction grammar. Minimal design gets weaker when every section tries
to be equally quiet. It gets stronger when the page has one clear structural
idea and every module obeys it.

### 22.1 Product Home Page Pattern

Use this when building a serious product homepage, especially for AI, data,
infrastructure, developer tools, operations, security, finance, analytics, or
enterprise workflows.

Structure:

1. top navigation with logo, 3-5 links, one primary action
2. hero with exact claim, short proof line, one primary CTA, one secondary link
3. product object or interface proof, not a decorative illustration
4. credibility row with customer names, metrics, standards, or integrations
5. three-part capability section with restrained cards or border-separated rows
6. workflow section showing before, during, after
7. conversion section with repeated CTA and one reassurance line

Visual rules:

- keep the nav height compact, usually 64-80px
- place the hero on an open canvas with no surrounding card
- allow one large type moment, then reduce scale quickly
- use a single accent for the primary CTA and active states
- avoid hero badges unless the badge contains concrete product proof
- make every icon 16-20px and visually aligned to the text baseline
- avoid mixed illustration styles
- use rows and dividers when cards feel too generic

Good hero anatomy:

```text
[small proof label]
Big exact outcome headline.
One sentence explaining the product's operating model.
[Primary action] [Secondary link]
Product proof object
```

Bad hero anatomy:

```text
[empty announcement badge]
Generic future-of-work headline.
Vague value proposition.
[Get started] [Learn more]
Floating decorative panels
```

Minimal homepage checklist:

- the product category is obvious in five seconds
- the user can understand what changes after adopting the product
- there is enough proof to justify the whitespace
- the CTA does not compete with other buttons
- the product visual contains readable, meaningful details
- section transitions use spacing, not decorative separators

### 22.2 SaaS Dashboard Pattern

Use this when designing the first logged-in screen for a minimal SaaS product.
Minimal dashboards should feel calm but not sparse. The user needs density,
prioritization, and control.

Layout recipe:

- left sidebar: 240-280px, off-white or very pale gray
- top bar: 56-64px, title, search, account, optional status
- main grid: 12 columns on desktop, 4 columns on tablet, 1 column on mobile
- primary panel: spans 7-8 columns
- secondary stack: spans 4-5 columns
- cards: 1px border, 6-8px radius, no strong shadow
- spacing: 16px between dense panels, 24px between major groups

Dashboard module hierarchy:

1. current state
2. next action
3. recent changes
4. comparative data
5. secondary settings

Tone:

- status language should be short and factual
- labels should be nouns, not marketing phrases
- empty states should tell the user what to do next
- errors should name the failed operation and recovery action

Color:

- neutral surfaces carry most of the interface
- accent only marks selected nav, primary action, and key data highlight
- warning and error colors should be muted but clear
- charts should use 1-3 colors maximum

Common dashboard mistakes:

- cards with identical visual weight
- decorative metric tiles without action
- weak gray text that cannot be read
- repeated blue buttons in every panel
- icons used as decoration rather than scanning aids
- charts with palette colors unrelated to state

### 22.3 Editorial Minimal Landing Pattern

Use this for founders, studios, consultancies, research labs, creative tools, and
brands where intellectual taste matters as much as product clarity.

The page should feel more like a designed publication than a generic SaaS page,
but still remain functional.

Structural moves:

- large serif or sharp display headline
- very narrow supporting copy, usually 520-680px
- generous top whitespace
- section labels in small uppercase or compact sans
- long-form text blocks balanced by precise product images or case panels
- editorial rhythm: large statement, proof, smaller annotation

Typography:

- display serif: 56-96px desktop, 40-56px mobile
- body: 17-19px, line-height 1.55-1.7
- captions: 12-13px, line-height 1.4
- labels: 11-12px, uppercase optional, letter spacing no more than 0.04em

Color:

- warm white, ink, parchment gray, one restrained accent
- avoid beige monotony by introducing crisp black, cool gray, or a precise color
- accent should appear in links, small rules, active filters, or one CTA

Interaction:

- hover states can be textual: underline, color shift, or subtle movement
- avoid springy motion
- avoid neon gradients
- transitions should feel like page turning or careful reveal

Use editorial minimalism when:

- the content has depth
- the brand has a point of view
- the product can tolerate slower pacing
- the page needs premium credibility

Do not use it when:

- the product needs immediate task density
- the audience needs technical documentation first
- the brand has no real content to support the spaciousness

### 22.4 High-Trust Enterprise Pattern

Use this for AI governance, security, compliance, procurement, financial tools,
healthcare software, and complex B2B systems.

Visual formula:

```text
quiet canvas + exact copy + restrained accent + proof modules + conservative motion
```

Required modules:

- trust navigation with security, documentation, pricing, contact
- proof strip with logos, certifications, uptime, or deployment facts
- architecture diagram or workflow proof
- role-based benefits
- security or governance block
- implementation reassurance

Palette:

- background: #F7F8FA, #FAFAF8, or #FFFFFF
- text: #111827, #0B0B0D, or #1F2937
- muted: #6B7280 or #667085
- border: #E5E7EB or #DADDE3
- accent: one of violet, blue, graphite, green, or amber

Enterprise minimal pages should not look playful by default. If personality is
needed, put it into typography, copy precision, and one intelligent visual
system, not into random color.

Trust details:

- use real UI screenshots whenever possible
- show integration logos in monochrome or low-saturation form
- include security facts close to conversion points
- make contact/sales actions visible but not pushy
- show implementation steps with concrete timeframes if available

### 22.5 Developer Tool Pattern

Minimal developer pages need faster scanning than general SaaS pages. The
interface should respect technical readers: precise labels, code blocks, direct
installation steps, clear docs path.

Essential modules:

- hero claim with product category
- installation snippet or API snippet above the fold
- docs link in nav and hero
- small feature grid using technical nouns
- code example with syntax highlighting
- integration/runtime support table
- changelog or GitHub/community signal

Visual language:

- monospace should appear intentionally, not everywhere
- code blocks should have strong contrast and readable line-height
- use thin borders and compact tabs
- avoid oversized marketing illustrations
- keep CTAs concrete: "Read docs", "Start building", "View examples"

Component rules:

- code block radius: 6-8px
- code font size: 13-15px
- line-height: 1.55-1.7
- tab height: 36-44px
- copy buttons: icon button with tooltip
- status chips: small, text-first, no decorative glow

Developer minimalism is allowed to be denser than consumer minimalism. Empty
space should organize information, not slow down access to technical proof.

### 22.6 Minimal Form Pattern

Forms are where minimal design often fails. A minimal form must be clearer than
a decorated form, not merely lighter.

Form anatomy:

- title names the operation
- helper text explains consequence, not obvious mechanics
- labels are always visible
- inputs have enough height and contrast
- error state appears near the field
- primary action is visually strongest
- secondary action is text or ghost button

Measurements:

- label size: 13-14px
- input text: 15-16px
- input height: 40-48px
- field gap: 14-18px
- group gap: 24-32px
- radius: 6-8px
- border: 1px solid neutral

States:

- hover: border darkens slightly
- focus: one accent ring or border, never multiple effects
- error: red border plus message
- disabled: lower contrast but still legible
- loading: preserve button width
- success: small confirmation, not a confetti moment

Minimal form anti-patterns:

- placeholders instead of labels
- borderless inputs on white background
- weak gray helper copy
- icon-only error states
- multiple primary buttons
- giant empty modals with tiny forms

### 22.7 Minimal Pricing Pattern

Pricing needs confidence. A minimal pricing page can be quiet, but it cannot be
ambiguous.

Recommended structure:

1. exact headline: "Pricing that scales with usage" or equivalent
2. billing toggle if needed
3. 3-4 plan cards or a two-column plan/comparison layout
4. primary plan emphasized by border, subtle tint, or label
5. feature comparison table
6. enterprise contact band
7. FAQ with short, plain answers

Plan card rules:

- one emphasized plan only
- price should be the largest element in the card
- CTA should sit in the same vertical position across cards
- feature lists should have comparable length
- avoid decorative illustrations inside pricing cards
- use check icons only when they improve scanning

Minimal pricing color:

- neutral base
- one accent for selected/recommended plan
- green only for confirmation, not every checkmark
- warning colors only for billing alerts

### 22.8 Minimal Case Study Pattern

Case studies in minimal design should read like evidence. Avoid turning them
into generic brand storytelling.

Structure:

- client/project title
- one sentence outcome
- key metrics
- problem/context
- intervention/system
- final result
- visual evidence

Visual rhythm:

- big title
- narrow intro
- metric row
- large image or UI proof
- alternating text and evidence blocks
- restrained pull quote if useful

Metric style:

- large numbers
- small uppercase labels or compact captions
- no decorative icons unless they identify metric type
- keep numbers aligned on a grid

### 22.9 Minimal Navigation Systems

Navigation should feel effortless. Minimal nav gets damaged by hidden complexity.

Top nav:

- use 4-6 visible items maximum
- avoid vague labels like "Platform" unless it opens a strong menu
- primary CTA on the right
- secondary CTA as text link if needed
- sticky only when it helps conversion or task use

Sidebar:

- use when the product has persistent sections
- keep icons same stroke weight
- active state should be obvious
- nested groups should be collapsible only when truly needed
- do not over-round nav pills

Mega menu:

- allowed for enterprise/product ecosystems
- use columns with clear headings
- include short descriptions only when they help selection
- avoid screenshot thumbnails unless they are useful

Mobile nav:

- compact header
- full-screen drawer or sheet
- large tap targets
- preserve primary CTA
- do not hide critical product category links

### 22.10 Minimal Data Visualization

Minimal charts should be readable first and beautiful second. The primary job is
comparison.

Rules:

- prefer direct labels over legends when possible
- gridlines should be pale and few
- axis labels should be legible
- use color for meaning, not decoration
- use one highlight color and neutral series for context
- annotate key moments directly
- avoid 3D charts, heavy gradients, and ornamental patterns

Chart palette:

```css
:root {
  --chart-primary: #111827;
  --chart-secondary: #9CA3AF;
  --chart-accent: #2563EB;
  --chart-positive: #059669;
  --chart-warning: #D97706;
  --chart-negative: #DC2626;
  --chart-grid: #E5E7EB;
}
```

For dashboards:

- line charts can use 2px strokes
- bars should have small radius or square ends
- selected states can use accent fill
- tooltips should be compact and high contrast
- avoid huge chart cards with tiny data

### 22.11 Minimal Motion Grammar

Motion in minimal design should clarify state and sequence.

Allowed:

- fade in for section reveal
- 8-16px translate for entering elements
- 120-220ms hover transitions
- 250-450ms page section reveal
- smooth height transition for accordions
- gentle tab indicator motion

Use sparingly:

- parallax
- scroll-linked progress
- text reveal
- product mockup movement

Avoid:

- bouncing CTAs
- random floating shapes
- constant background motion
- hover effects that move layout
- loading animations that feel playful in serious products

Motion tokens:

```css
:root {
  --ease-minimal: cubic-bezier(0.16, 1, 0.3, 1);
  --duration-fast: 140ms;
  --duration-base: 220ms;
  --duration-slow: 420ms;
}
```

### 22.12 Minimal Brand Adaptation Matrix

Use this matrix to adapt minimal design without making every page look the same.

| Brand Need | Add | Avoid |
| --- | --- | --- |
| authority | larger type, strict grid, proof diagrams | random badges |
| warmth | off-white canvas, softer radius, human copy | beige overload |
| precision | black/white contrast, hairlines, compact spacing | decorative gradients |
| innovation | one vivid accent, abstract product logic | neon everywhere |
| luxury | serif type, large margins, restrained imagery | stock glamour |
| technical depth | code, diagrams, tables | vague illustration |
| trust | certifications, exact claims, conservative palette | playful motion |
| speed | dense modules, clear CTAs | oversized editorial pacing |

### 22.13 Minimal Section Recipes

Hero section:

```text
max-width: 1120-1280px
padding-top: 96-144px desktop
headline width: 720-980px
support copy width: 520-680px
CTA row gap: 12px
proof object margin-top: 48-72px
```

Feature section:

```text
section padding: 88-128px
intro width: 560-720px
grid: 3 columns desktop, 1 mobile
card padding: 24-32px
icon size: 18-24px
```

Proof section:

```text
background: slightly tinted neutral or white
top border optional
logo opacity: 0.65-0.8
metric text: 40-64px
caption: 13-15px
```

Conversion section:

```text
centered or split with product proof
headline: smaller than hero
copy: direct reassurance
primary CTA only, secondary link optional
no new visual language
```

### 22.14 Minimal Copy Rules

Minimal visual systems require precise language. Weak copy makes whitespace feel
empty.

Use:

- concrete nouns
- active verbs
- measurable outcomes
- product-specific terminology
- clear audience signals
- short CTA labels

Avoid:

- "supercharge"
- "unlock your potential"
- "next-generation platform"
- "seamless experience"
- "all-in-one solution" unless literally true
- vague innovation claims

Better patterns:

| Weak | Stronger |
| --- | --- |
| Build faster with AI | Review contracts in minutes |
| The future of work | Your support queue, resolved automatically |
| Unlock insights | See which accounts changed this week |
| Scale your team | Route every lead to the right owner |
| Make better decisions | Compare forecast risk by region |

### 22.15 Minimal QA Checklist

Run this before considering a minimal page finished.

Visual hierarchy:

- can a user identify the primary action instantly?
- does the headline dominate only where it should?
- are section headings consistently scaled?
- is body text strong enough to read comfortably?
- do cards have differentiated purpose?

Spacing:

- are related elements close enough?
- are unrelated sections separated enough?
- does mobile spacing preserve hierarchy?
- are there accidental gaps from layout defaults?

Color:

- is there exactly one main accent?
- are neutrals varied enough to show structure?
- does the page avoid a one-note palette?
- do warning/error/success states remain distinct?

Content:

- is there concrete proof above or near the fold?
- does every section have a job?
- are labels specific?
- are empty states useful?

Interaction:

- are focus states visible?
- do hover states respect layout stability?
- do disabled states remain legible?
- is motion purposeful and quiet?

### 22.16 Minimal Debugging Guide

If the page feels empty:

- add proof, not decoration
- strengthen headline specificity
- introduce product screenshots or diagrams
- tighten spacing between related elements
- add section labels or rules

If the page feels generic:

- change type pairing
- define a sharper accent role
- replace stock icons with product-specific diagrams
- reduce card repetition
- add real workflow language

If the page feels cold:

- warm the canvas slightly
- improve copy tone
- use human-scale imagery
- soften radius modestly
- add one warm neutral, not a full beige palette

If the page feels cluttered:

- remove duplicate CTAs
- collapse secondary text
- reduce icon count
- convert cards into rows
- make dividers lighter

If the page feels inaccessible:

- darken body text
- enlarge input labels
- increase tap targets
- strengthen focus rings
- simplify color-coded meaning

### 22.17 Minimal Implementation Guardrails

CSS guardrails:

- define tokens before components
- use semantic variables, not raw colors everywhere
- set max widths deliberately
- use clamp only for layout dimensions, not uncontrolled type scaling
- preserve line-height across breakpoints
- test at 360px, 768px, 1280px, and 1440px

Component guardrails:

- buttons should not change size on hover
- cards should not shift layout on hover
- badges should not wrap awkwardly
- tables should have horizontal strategy on mobile
- icons should align with text
- long labels should not overflow controls

Design guardrails:

- no random decorative blobs
- no gradient backgrounds unless the style source supports them
- no nested cards
- no card-heavy page sections by default
- no weak placeholder text
- no inaccessible gray-on-gray interfaces

### 22.18 Minimal Skill Usage Notes For Agents

When an agent uses this skill, it should not merely apply a white background and
black text. It should make explicit design choices:

1. choose the archetype
2. choose the type personality
3. choose the accent role
4. choose the surface grammar
5. choose the page rhythm
6. choose proof modules
7. choose interaction states
8. audit against anti-slop rules

The highest-value output is not "minimal-looking"; it is a system where every
visual decision reduces cognitive load while increasing trust.

## 23. QUICK REFERENCE

### Choose The Minimal Type

| Need | Use |
| --- | --- |
| AI authority | Architectural Minimal Authority |
| warm B2B SaaS | White Canvas Thoughtful Function |
| exacting precision | High-Contrast Precision Blueprint |
| finance/data clarity | Analytical Blueprint |
| enterprise conversion | Strategic Polished Steel |

### Do

- use one accent
- keep body text readable
- define type roles
- show product proof
- use spacing as hierarchy
- choose a surface system
- keep interaction states clear

### Don't

- confuse minimal with empty
- overuse cards
- add decorative colors
- hide CTA
- use low-contrast gray text
- rely on default typography
- add shadows everywhere

### Build Order

1. Define product goal.
2. Choose minimal archetype.
3. Define neutral palette.
4. Define one accent role.
5. Define type system.
6. Define spacing/radius.
7. Build hero with clear CTA.
8. Add product proof.
9. Add sections with consistent rhythm.
10. Audit contrast and hierarchy.

### Final Taste Test

If removing decoration makes the page clearer, you are moving toward minimal
design. If removing decoration makes the page confusing, you have not replaced
it with hierarchy yet.
