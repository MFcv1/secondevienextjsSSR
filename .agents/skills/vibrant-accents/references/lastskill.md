# Vibrant Accents

## Mandatory `<design_plan>`

Before substantial UI code, output a compact `<design_plan>` block. Include:

1. **Use case:** page/app type, audience, primary action, emotional target.
2. **Style direction:** one Vibrant Accents archetype below.
3. **Operating mode:** density, motion, decoration, contrast, radius, and asset burden.
4. **First viewport:** nav type, H1 width/line strategy, neutral product proof with a governed vivid action/category accent, CTA treatment, next-section hint.
5. **System contracts:** type, color, surface, radius, spacing, depth, state, and motion tokens.
6. **Component plan:** at least four concrete Vibrant Accents components with states.
7. **Motion plan:** accent sweep or category transition, timing, performance guardrail, and reduced-motion fallback.
8. **Anti-slop sweep:** top three failure modes for this style and how you will avoid them.

If the request is tiny, do this mentally and keep the final answer concise.


## Core Directive

Create color energy with strict governance. Vibrant Accents is not a colorful theme; it is a system that gives every saturated color a narrow job. The interface should feel clean, energetic, vivid, modern, selective, and confident while preserving readable typography, clear product proof, stable components, and obvious actions.

Use vibrant accents when a product needs stronger conversion focus, a memorable action color, category differentiation, campaign energy, optimistic AI/product mood, or B2B polish with one vivid signal.


## Style Operating Mode

| Control | Setting |
| --- | --- |
| density | Medium. Clear neutral layout with vivid focal points. |
| motion | Medium. Accent sweep, hover fill, category transition, chart highlight. |
| decoration | Medium but role-based. |
| contrast | Neutral base with high-saturation accents. |
| radius | 8-20px depending on product; 999px for chips. |
| type | Modern sans with confident display weights. |
| assets | Product screens, brand visuals, charts, portfolio images, color-coded categories. |

## Signature System

- Accent Budget: one primary action color, one secondary category color, one semantic alert/success system.
- Neutral Field: white, black, gray, or muted substrate lets accent color carry hierarchy.
- Vivid Interaction: hover/focus can introduce accent fills, but resting state stays disciplined.
- Category Encoding: colors can group marketplace/product categories if the mapping is consistent.

## Differentiation

Use Vibrant Accents when brands needing restrained layouts with vivid highlights, SaaS, portfolios, consumer pages, launch sites. If removing the neutral product proof with vivid accent role, token rules, or signature components leaves a generic page, this skill is the right lens and the signature object must stay. Use `expressive-brand` when voice/motif is the identity; use this when governed vivid color is the main differentiator.
## Core Principles

- Bright color must be either action, active/focus, semantic state, brand marker, section surface, category classification, product atmosphere, or image/illustration support.
- If a color has no role, remove it.
- The neutral anchor is mandatory. Vivid color needs a calm field to matter.
- Body copy should almost never be saturated.
- One primary action color owns conversion.
- Supporting accents cannot compete with primary action.
- Gradients should frame product meaning or atmosphere, not replace layout.
- Typography must be strong enough to stand beside color.

## Big Color Vs Small Color

### Big Color

Large backgrounds, blocks, campaign panels, hero gradients, major cards.

Use for:

- playful or campaign energy
- product category zones
- hero atmosphere
- large visual interruption
- photo/illustration staging

Requirements:

- Strong text contrast.
- Limited number of major color surfaces.
- Clear section boundaries.
- Short copy on the color, never long paragraphs.
- Neutral reset after the color moment.

Examples:

- SquadEasy amber/violet blocks.
- Stripe contained hero/product gradients.
- Base44 pastel workspace gradient.

### Small Color

Buttons, outlines, links, active states, focus rings, tags, charts, step indicators.

Use for:

- conversion
- selected/current state
- navigation focus
- process steps
- chart highlights
- category chips

Examples:

- Empower yellow CTA.
- Copy violet process/CTA.
- Base44 lime border.
- Stripe green/orange micro edges.

Rule: do not casually promote a small action color into a large background. Do not demote a large section color into many tiny buttons if it will confuse action hierarchy.

## Accent Budget

Use one of these budgets:

| Budget | Neutral | Main Accent | Support | Gradient/Big Color | Use |
| --- | --- | --- | --- | --- | --- |
| Refined Product | 70% white/navy | 8% violet | 2% green/orange edges | 5% contained gradient | fintech, SaaS, devtools |
| Playful Block | 40% warm color | 10% lime action | 10% pink/blue/green | 20% violet/amber blocks | campaign, community |
| Soft AI | 75% pearl/white | 6% lime | 4% orange | 10% pastel gradient | AI builder, creative workspace |
| Dark Finance | 55% dark | 8% yellow | 7% muted yellow | 25% light sections | finance, banking |
| Enterprise Violet | 70% pale surfaces | 8% violet | 5% violet tints | minimal | B2B process/product |

Audit every bright color:

1. What is its name?
2. What is its role?
3. Where is it allowed?
4. Where is it forbidden?
5. Does it compete with the CTA?
6. Does it pass contrast where text appears?
7. Does it appear often enough to be memorable but rarely enough to matter?

## First Viewport Protocol

The first viewport must feel vibrant without losing product clarity.

- Use one neutral base or one big color field, not a pile of competing backgrounds.
- H1 names product category, offer, or literal user outcome.
- Primary CTA uses the main action color.
- Secondary action uses outline, neutral fill, or action-soft token.
- Proof object is visible: product panel, chart, dashboard, photo cutout, workflow, category grid, process map.
- Gradients and photo cutouts cannot obscure text or controls.
- Next-section hint should reveal product proof, process, categories, or trust.
- Mobile reduces big color complexity before reducing text legibility.

## Component Arsenal

Use at least four for full-page work:

### Refero Expansion Component Deltas

- Accent budget is one dominant vivid role plus semantic states: Visitors violet action, Tella violet hero/action, Programa acid note only, Revenue-Grade violet ops CTA, Speakeasy spectrum divider only.
- Component radii must not drift: Visitors 16/24/9999px, Tella 24/32/40px, Programa 10/16px, Revenue-Grade 10/20/1425.6px. Use the source radius to distinguish brand accent from generic SaaS.
- Vivid colors must sit on neutral fields with readable text; never place low-contrast text on lavender, acid yellow, or spectrum gradients without a protected panel.

Motion should reveal, select, focus, or show progression. It should not turn color into noise.

| Motion | Use | Properties | Reduced Motion |
| --- | --- | --- | --- |
| Accent fill sweep | buttons, tabs, cards | transform on pseudo-element | immediate fill/border |
| Gradient drift | hero/product backdrop | background position or transform layer | static gradient |
| Chart highlight | data proof | transform/opacity/stroke-dashoffset | final highlighted state |
| Category switch | filters/tabs | opacity + transform under 220ms | instant content swap |
| Process pulse | current step only | outline/box-shadow | static active border |

}

@keyframes va-gradient-drift {
  from { background-position: 0% 20%; }
  to { background-position: 100% 80%; }

  stroke-dasharray: 1;
  stroke-dashoffset: 1;
  animation: va-chart-draw 900ms cubic-bezier(.2, 0, 0, 1) forwards;

## Motion System

- Vibrant accent motion is short and color-aware: 120-220ms for hover/focus color, 180-320ms for accent rail/indicator movement, no rainbow loops. Reduced motion keeps the color state as a static border/fill.

## Absolute Bans

- No different color theme in every section.
- No vivid accents as body text.
- No accent variety that makes states ambiguous.
- No raw Tailwind typography, spacing, radius, color, or shadow defaults when a style token exists.
- No generic centered hero without the style's required proof/media/type object.
- No status colors without semantic role mapping and visible text.
- No component states left implicit: include hover, focus-visible, selected, loading, empty, error, success where relevant.

## States

| State | Treatment |
| --- | --- |
| Hover | accent edge, fill sweep, slight lift |
| Focus | high-contrast ring tied to action or category |
| Active | filled state or pressed border |
| Selected | fill + label/icon + border, not color alone |
| Loading | neutral skeleton with accent progress edge |
| Empty | neutral surface with one vivid next action |
| Error | semantic red/brown separate from brand palette |
| Success | semantic green separate from decorative green unless palette defines it |
| Disabled | neutral muted surface; do not use accent |

Semantic state colors:

- Keep them separate from brand accents unless the project explicitly maps them.
- Do not use decorative orange/pink for errors unless contrast and meaning are clear.
- Do not use green as both success and decorative highlight inside the same component cluster without labels.

## Accessibility

- No saturated body text.
- No yellow/lime buttons with white text.
- Long paragraphs sit on neutral or very pale backgrounds.
- Large color blocks need short copy and dark text.
- Focus states must be visible on both neutral and colorful surfaces.
- State must not be color-only.
- Gradients cannot sit behind body text unless contrast is tested at every stop.
- Motion has reduced-motion fallback.

## Implementation Workflow

1. Pick source archetype.
2. Name the neutral anchor.
3. Name primary action color and forbidden roles.
4. Decide big color or small color.
5. Define support accents with narrow jobs.
6. Build first viewport with proof and CTA.
7. Build four components and implement states.
8. Add accent sweep or highlight motion.
9. Audit every bright color.
10. Test mobile and reduced motion.

## Anti-Slop

Reject:

- Random rainbow palettes.
- All buttons different colors.
- Vivid accents as body text.
- Gradients on every section.
- Semantic colors used decoratively.
- Generic blue CTA unless blue is the brand.
- Glow effects on all components.
- Inaccessible text on color.
- Childish color blocks for serious products.
- Enterprise pages with playful confetti.
- CTA and decoration competing at same visual weight.

Correct:

- If too colorful, lower support accents to edges/tags and restore neutral space.
- If too generic, choose a named palette and enforce its radius/type rules.
- If CTA is weak, remove competing accents near it.
- If gradients are noisy, contain them inside hero/product panels.
- If color meaning is ambiguous, add labels, icons, or text state.
- If mobile collides, remove big color overlays and simplify proof composition.

## Pre-Output Checklist

- First viewport contains a real neutral product proof with vivid accent role.
- One Vibrant Accents archetype is clearly dominant.
- Execution tokens are declared and component CSS uses them.
- Typography uses named pairings, not raw Tailwind defaults.
- Spacing uses `--s-*` or style tokens, not mixed arbitrary padding.
- Radius, depth, and state colors use the token contract.
- Status labels use role mapping plus `--status-{role}-bg/fg`.
- Components include hover, focus-visible, selected, loading, empty, error, and success where relevant.
- Motion maps to accent sweep/category transition and has a reduced-motion fallback.
- Mobile layout preserves the style without overflow, unreadable text, or hidden controls.
