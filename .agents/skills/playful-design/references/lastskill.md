# Playful Design

## Mandatory `<design_plan>`

Before substantial UI code, output a compact `<design_plan>` block. Include:

1. **Use case:** page/app type, audience, primary action, emotional target.
2. **Style direction:** one Playful Design archetype below.
3. **Operating mode:** density, motion, decoration, contrast, radius, and asset burden.
4. **First viewport:** nav type, H1 width/line strategy, tactile progress, chunky product card, reward state, or playful onboarding proof, CTA treatment, next-section hint.
5. **System contracts:** type, color, surface, radius, spacing, depth, state, and motion tokens.
6. **Component plan:** at least four concrete Playful Design components with states.
7. **Motion plan:** press, bounce, progress, or reward transition, timing, performance guardrail, and reduced-motion fallback.
8. **Anti-slop sweep:** top three failure modes for this style and how you will avoid them.

If the request is tiny, do this mentally and keep the final answer concise.


## Core Directive

Make the interface feel joyful, tactile, approachable, and alive while keeping the next action obvious. Playful design is not childish decoration. It is a controlled system of color roles, rounded geometry, physical button behavior, character or object vocabulary, generous spacing, friendly copy, and feedback that helps users understand progress.

Use playful design for learning apps, consumer brands, creative tools, family products, wellness and community pages, product launches, games-adjacent experiences, and B2B SaaS that needs warmth without losing credibility.


## Style Operating Mode

| Control | Setting |
| --- | --- |
| density | Medium. Use chunky modules and clear steps. |
| motion | Medium-high. Press, bounce, success, drag, gamified progress. |
| decoration | High but controlled by one metaphor. |
| contrast | Bright colors with dark readable text and disciplined neutral support. |
| radius | 24-48px for cards, 999px for pills, tactile shadows. |
| type | Rounded or bold friendly sans; display type can be large. |
| assets | Illustrations, product objects, stickers, icons, progress visuals, playful photos. |

## Signature System

- Playfulness Control Knobs: color intensity, radius, shadow depth, motion bounce, character presence.
- Tactile Button System: press states with bottom shadow, slight translate, and clear label.
- Gamified Progress: steps, streaks, levels, or rewards must map to real user progress.
- Mature Clarity: playful surfaces still need normal forms, readable prices, and accessible focus.

## Differentiation

Use Playful Design when playful brands, education, consumer apps, creative tools, family products, gamified experiences, friendly SaaS. If removing the tactile progress/product object, token rules, or signature components leaves a generic page, this skill is the right lens and the signature object must stay. Use `pastel` for calmer softness; use this when progress, tactile controls, and playful feedback improve comprehension.
## Playfulness Principles

- Playfulness must reduce intimidation, explain progress, invite exploration, or make the brand memorable.
- Every bright color needs a job. Illustration can use more color than UI chrome.
- Every tactile control needs a real pressed, hover, focus, disabled, loading, error, and success state.
- Characters, stickers, badges, and confetti must map to product meaning or user progress.
- Sensitive domains need warmth, not jokes. Wellness, finance, health, education, and family support require calm copy and clear trust.
- Mature playful systems often use neutral bases with color in modules rather than saturation everywhere.

## Maturity Scale

Use the maturity scale to avoid childish output.

| Level | Best For | Visual Moves | Keep Calm |
| --- | --- | --- | --- |
| Gentle Play | wellness, family support, onboarding | rounded cards, warm illustrations, slow motion | errors, privacy, practitioner credentials |
| Active Play | learning, habits, creator tools | progress path, tactile controls, rewards | account, billing, destructive actions |
| Product Play | hardware, launches, games | iconic color, physical buttons, product render | specs, price, support |
| Professional Play | B2B SaaS, productivity | neutral base, colorful modules, friendly empty states | pricing, security, tables, admin flows |
| Campaign Play | events, community, brands | large color blocks, photos, stickers | forms, legal, conversion CTA |

## Color Governance

Every playful palette needs roles:

```txt
canvas
primary text
primary action
secondary action/link
card accent surfaces
illustration/sticker colors
border/support neutrals
semantic success/warning/error
focus ring
disabled state
```

Rules:

- One primary CTA color per experience.
- Secondary colors can live in stickers, illustrations, cards, or category chips.
- UI state colors are not decorative colors.
- Text on saturated color needs explicit contrast testing.
- Avoid using a broad illustration palette for small functional controls.
- Use black/white or warm off-white as the neutral anchor when the palette is loud.

Safe audience pairings:

- Kids/family: yellow, green, blue, pink; characters welcome; high contrast mandatory.
- Adult wellness: warm yellow/orange with teal or blue support; slower motion.
- Creative tools: warm canvas, one vivid CTA, gradients in hero or empty states.
- B2B: black/white base, controlled violet/blue, colorful modules only.
- Hardware/gaming: iconic brand color, tactile contrast, concise copy.

## Tactile Button System

Buttons are the signature control in playful design. They need a physical contract.

### Button Anatomy

- Top fill: primary action color.
- Bottom edge/shadow: darker sibling or ink.
- Press distance: 2-6px depending on style.
- Radius: archetype-specific.
- Border: only if needed for contrast or arcade feel.
- Label: clear verb, not cute ambiguity.
- Focus: visible ring outside the physical shadow.
- Disabled: keep shape, reduce saturation, no press.
- Loading: label changes without width shift.

TSX:

```tsx
type PlayButtonState = "idle" | "loading" | "success" | "error" | "disabled";

export function TactileButton({
  state = "idle",
  children = "Start"
}: {
  state?: PlayButtonState;
  children?: React.ReactNode;
}) {
  const text = {
    idle: children,
    loading: "Working",
    success: "Done",
    error: "Retry",
    disabled: children
  }[state];

  return (
    <button
      className="tactile-button"
      data-state={state}
      disabled={state === "disabled"}
      aria-busy={state === "loading"}
      type="button"
    >
      <span>{text}</span>
    </button>
  );
}
```

## Chunky Shadows

Chunky shadows should feel physical, not blurred.

Use:

- Solid offset shadows on buttons and selectable cards.
- Darker sibling color for playful action controls.
- Ink shadow for arcade or poster-like buttons.
- 2-4px offset in dense UI, 6-10px for marketing hero CTAs.

Avoid:

- Large blurred shadows on every card.
- Shadows that imply elevation but have no interactive meaning.
- Shadow colors that create low contrast or muddy edges.
- Hover shadows that resize layout.

CSS:

```css
.chunky-card {
  border: 2px solid var(--play-ink);
  border-radius: 24px;
  background: #ffffff;
  box-shadow: 8px 8px 0 var(--play-ink);

  transition: transform 160ms var(--play-bounce), box-shadow 160ms ease;

  transform: translate(-2px, -2px) rotate(var(--tilt, -.3deg));
  box-shadow: 10px 10px 0 var(--play-ink);
}

.chunky-card:active,

  transform: translate(4px, 4px);
  box-shadow: 3px 3px 0 var(--play-ink);
```

## Character And Sticker System

Use character or sticker assets only when they have a defined role.

### Mature Character Rules

- Character has a job: guide, status, reward, empty state, product metaphor, or friendly support.
- Anatomy is consistent: head/body shape, eyes, limbs, stroke, fill palette, mouth states.
- Emotional states map to product states: neutral, thinking, success, error, waiting, locked, empty.
- Character never blocks text, buttons, pricing, forms, product specs, or screenshots.
- Character appears small in B2B and support contexts; large only in consumer/learning hero.
- Copy around characters stays plain and useful.

### Sticker Rules

- Stickers are modular marks for category, reward, annotation, or proof.
- Use a finite sticker set: 6-10 shapes max for a page.
- Stickers align to a grid or safe-zone, even if they look casual.
- Stickers can rotate by `-4deg` to `4deg`; do not randomize wildly.
- Stickers need alt text if meaningful; decorative stickers are `aria-hidden`.

### Refero Expansion Component Deltas

- Playful controls need tactile but stable dimensions: Navigate buttons 25.146px and cards 43.2px, Cycle 10/20/100px, Geniestudio 32/90px, Moderne 4/15px, PostHog 4/8/9999px. Set fixed min-heights so hover states do not resize.

## Spring Physics

Use spring-like motion for tactile actions, not for every scroll reveal.

- Refero playful packs support tactile motion at 160-260ms for press/hover and 260-420ms for reward/progress feedback. Use spring only on buttons, chips, draggable controls, or reward states; disable scale and bounce under `prefers-reduced-motion`.

Timing tokens:

```css
:root {
  --play-snap: cubic-bezier(.2, 0, 0, 1);
  --play-spring-soft: cubic-bezier(.34, 1.56, .64, 1);
  --play-spring-firm: cubic-bezier(.25, 1.3, .35, 1);
  --play-fast: 120ms;
  --play-medium: 220ms;
  --play-slow: 420ms;
}
```

Motion roles:

- Press: 120ms, direct, paired with bottom shadow.
- Pop: 220ms, small overshoot for selection/reward.
- Reveal: 420ms, soft entrance for hero objects or cards.
- Drag: transform-only, no layout changes.
- Reward: one brief celebration after meaningful completion.

CSS:

```css
[data-play-motion="pop"] {
  animation: play-pop 260ms var(--play-spring-soft) both;
}

@keyframes play-pop {
  0% { transform: scale(.92); opacity: .2; }
  70% { transform: scale(1.04); opacity: 1; }
  100% { transform: scale(1); opacity: 1; }
}

[data-play-motion="float"] {
  animation: play-float 5s ease-in-out infinite;
}

@keyframes play-float {
  0%, 100% { transform: translateY(0) rotate(-1deg); }
  50% { transform: translateY(-8px) rotate(1deg); }
}

[data-play-motion="reward"] {
  animation: play-reward 640ms var(--play-spring-soft) both;
}

@keyframes play-reward {
  0% { transform: translateY(8px) scale(.9) rotate(-4deg); opacity: 0; }
  45% { transform: translateY(-8px) scale(1.08) rotate(2deg); opacity: 1; }
  100% { transform: translateY(0) scale(1) rotate(0); opacity: 1; }
}

@media (prefers-reduced-motion: reduce) {
  [data-play-motion] {
    animation: none !important;
    transition-duration: .01ms !important;
    transform: none !important;
  }
  [data-play-motion="reward"] {
    box-shadow: 0 0 0 4px currentColor;
  }
}
```

Reduced motion equivalents:

- Press becomes inset border.
- Pop becomes color/ring change.
- Progress fill snaps to final width.
- Character expression changes statically.
- Reward toast appears without movement.

## State Language

Use playful state design without reducing clarity.

| State | Treatment |
| --- | --- |
| Empty | friendly illustration or sticker + one useful next action |
| Loading | skeleton with rounded rhythm; label says what is happening |
| Success | small reward pop, green/highlight, concrete confirmation |
| Error | warm but clear message, recovery action, no jokes |
| Disabled | same shape, muted fill, explanation if blocked |
| Selected | fill/ring/pressed state, not only color |
| Focus | obvious ring, keyboard parity with hover |

## Accessibility

- Contrast must pass on every colored surface.
- Do not place long paragraphs on saturated blocks.
- Touch targets stay at least 40px, preferably 44px+.
- State is not color-only.
- Character and sticker assets do not cover controls or text.
- Avoid motion during typing-heavy tasks.
- Respect reduced motion for all bouncing, floating, reward, drag, and scroll effects.
- Use recognizable icons before making them charming.

## Anti-Slop

Reject:

- Random rainbow palettes.
- Babyish tone for adult users.
- Mascots unrelated to product.
- Confetti or badges for routine clicks.
- Every component bouncing.
- Product screenshots hidden behind art direction.
- CTA copy that sounds cute but unclear.
- Inconsistent radii and shadow depths.
- Color-only learning feedback.
- Trust-sensitive content in overly playful cards.

Correct:

- If too childish, reduce illustration density, ground with black/white or warm neutral, and make copy more direct.
- If too corporate, add tactile controls, friendly empty states, one visual metaphor, and clearer reward moments.
- If noisy, assign each color one job and remove simultaneous decorations.
- If flat, add a real tactile button system, chunky selected state, and a relevant object/character.
- If inaccessible, darken text, move copy off saturated surfaces, add labels beyond color, and reduce motion.

## Absolute Bans

- No childish treatment for adult products.
- No random mascot, badge, or confetti without a job.
- No playful color that reduces readability.
- No raw Tailwind typography, spacing, radius, color, or shadow defaults when a style token exists.
- No generic centered hero without the style's required proof/media/type object.
- No status colors without semantic role mapping and visible text.
- No component states left implicit: include hover, focus-visible, selected, loading, empty, error, success where relevant.


## Pre-Output Checklist

- First viewport contains a real tactile progress/product object.
- One Playful Design archetype is clearly dominant.
- Execution tokens are declared and component CSS uses them.
- Typography uses named pairings, not raw Tailwind defaults.
- Spacing uses `--s-*` or style tokens, not mixed arbitrary padding.
- Radius, depth, and state colors use the token contract.
- Status labels use role mapping plus `--status-{role}-bg/fg`.
- Components include hover, focus-visible, selected, loading, empty, error, and success where relevant.
- Motion maps to press/bounce/reward transition and has a reduced-motion fallback.
- Mobile layout preserves the style without overflow, unreadable text, or hidden controls.
